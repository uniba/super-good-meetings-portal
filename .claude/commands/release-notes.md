# リリースノート生成スキル

SuperGoodMeetings Portal の `_release_notes/` 配下にリリースノートの Markdown ファイルを生成する。

## 前提

- 対象リポジトリは `super-good-meetings-portal`
- このスキルは2つの起動位置から呼ばれうる。**最初に作業ディレクトリ（以下 `$RN_DIR`）を決定する**：
  - カレントに `_release_notes/` が存在する → portal リポジトリ直下で起動。`$RN_DIR = _release_notes`
  - カレントに `portal/_release_notes/` が存在する → `super-good-meetings`（親リポジトリ）から submodule 越しに起動。`$RN_DIR = portal/_release_notes`、かつ commit/push 等の git 操作は `portal/` ディレクトリ（submodule）に対して行う
  - どちらも存在しない → ユーザーに portal リポジトリのパスを確認する
- 以降の手順で `_release_notes/` と書かれている箇所は、上で決めた `$RN_DIR` に読み替える

## 手順

### STEP 1: 議事録から変更内容を取得（SuperGoodMeetings MCP）

変更内容は、スプリントレビュー会議体のミーティング内にある「リリースアイテムを決定する」アジェンダの**議事録（`proceedings`）**から取得する。ユーザーに変更内容を貼り付けてもらう必要はない。

#### 1-1. 直近のミーティングを特定する

- 会議体（track）ID は `4b6b0265133f44f5aa8363436d7e0f09`（スプリントレビュー）
- `list_track_meetings` で一覧を取得する。**表示順（古い順）で返り、`total_count` で総数が分かる**ため、末尾（最新側）を `offset = total_count - 5` 程度で取得する
- 一覧には**未開催の予定ミーティングも含まれる**（`date` が未来の回）。さらに予定回は `meeting_items` が空のこともある
- 最新側から順に `get_meeting` で中身を確認し、**「リリースアイテムを決定する」アジェンダ（`what == "リリースアイテムを決定する"`）の `proceedings` が空でない最初の回**を「直近のミーティング」とする
- 候補が見つかったら、その回の `title`・`date` をユーザーに伝えて取得元を明示する

#### 1-2. 議事録を抽出してリリースノートのセクションに対応させる

対象アジェンダの `proceedings` を読み取る（`document` フィールドは空のテンプレ枠なので使わない）。`proceedings` 内の見出しを、以下のルールでリリースノートのセクションに振り分ける：

| 議事録の見出し | リリースノートのセクション |
| --- | --- |
| `新機能` | **`## 機能改修` に統合**（機能改修の項目と合わせて1つにまとめる） |
| `機能改修` | `## 機能改修` |
| `不具合修正` | `## 不具合修正` |
| `フィーチャーフラグ` | **含めない**（未公開機能のため除外する） |
| `プロダクトオーナー定金からひとこと` | `## プロダクトオーナー定金からひとこと`（STEP 5 で扱う） |

- 議事録は箇条書き（`-` 始まり・常体のこともある）で書かれている。リリースノートでは敬体・完了形（「〜になりました」「〜を修正しました」）に整え、各項目を `* ` で始める（STEP 6 の体裁に合わせる）
- 「プロダクトオーナー定金からひとこと」が空（`-` のみなど）の場合は、STEP 5 で変更内容から草案を作成する
- リリース日は、省略された場合は本日の日付を使用する旨を伝えて確認する

#### 1-3. 取得結果の確認

抽出・振り分けした変更内容の一覧をユーザーに提示し、過不足や分類の誤りがないか確認してもらう。分類が曖昧な項目があればユーザーに確認する。確認が取れてから次のステップに進む。

### STEP 2: 直近のリリースノートを参照

`$RN_DIR` 配下で最新のファイルを1〜2件読み、以下を踏襲する：

- フロントマターのフォーマット
- 見出し構成とセクションの並び順
- 箇条書きの文体（敬体、「〜になりました」「〜を修正しました」などの完了形）
- 最後の「プロダクトオーナー定金からひとこと」セクションの語り口

最新ファイルは `ls -1 "$RN_DIR" | sort | tail -2` で特定できる。

### STEP 3: タイトルの提案と確認

変更内容をもとにタイトル候補を **2つ** 提案し、ユーザーに選択または編集してもらう。

提案の方針：

- 大型アップデートがある場合：主要な変更を前面に出した長めのタイトル
  - 例: `プロジェクト画面が刷新されました！ロードマップにイベントとフェーズの両方を作成できるようになりました。その他、機能改修・不具合修正を行いました`
- 小規模な回：テーマを絞った短めのタイトル
  - 例: `アジェンダバスケットを中心とした機能改修・不具合修正を行いました`

タイトルが確定するまで次のステップに進まない。

### STEP 4: アイコンの決定

- デフォルトは `✨`
- ユーザーが明示的に別のアイコンを指定した場合のみ上書きする
- 確認は求めない

### STEP 5: 「プロダクトオーナー定金からひとこと」の草案作成

このセクションは毎回掲載する。プロダクトオーナー「定金」さんからの語りかけ口調でまとめる。

- 柔らかい語り口（「皆さん、〜されていますか？」「ぜひ試してみてください」など）
- 今回の目玉機能の使い方や、ユーザーへの呼びかけを含める
- 2〜4 項目の箇条書きで構成する

ユーザーが内容を提供していない場合は、変更内容から推測して **草案** を提示し、ユーザーに確認・編集してもらう。

### STEP 6: ファイルの生成

以下の形式で `$RN_DIR/YYYYMMDD.md` に書き込む：

```markdown
---
title: "<確定したタイトル>"
date: "YYYY.M.D"
icon: <アイコン>
---

[docs]

## 大型アップデート

* <項目>

## 機能改修

* <項目>

## 不具合修正

* <項目>

## プロダクトオーナー定金からひとこと

* <項目>
```

**書き込み時の注意：**

- ファイル名は `YYYYMMDD.md`（例: `20260408.md`）
- フロントマターの `date` は `"YYYY.MM.DD"` 形式（**月・日をゼロ埋めする**。例: `2026.04.08`、`2026.10.07`）。文字列ソートで正しく並ぶようにするため。
- 本文冒頭に `[docs]` を置く（ドキュメントへの参照プレースホルダ）
- 該当する変更がないセクションは見出しごと省略する（例: 大型アップデートがない回は `## 大型アップデート` セクションを丸ごと削除）
- 各項目は `* ` で始める（ハイフンではなくアスタリスク）

### STEP 7: 確認・報告

- 生成したファイルのパスを報告する
- ブランチ運用（`devel` 起点で作業、PR で `master` にマージ）は README に記載があるため、必要に応じて案内する
- コミット・プッシュはユーザーの指示があるまで行わない
- `super-good-meetings` から submodule 越しに編集した場合、portal リポジトリ（`portal/`）側で commit/push したうえで、親リポジトリの submodule ポインタ更新が別途必要になる旨を案内する

### STEP 8: Slack 告知メッセージの下書き作成（任意）

リリースが本番（`master`）に反映されたあと、社内告知用の Slack メッセージを作成する。ユーザーから依頼があった場合に実施する。

- **投稿先チャンネル**: `#pjsbiz-info`（channel_id: `C01T1GVHYKX`）
- このチャンネルは**外部共有（Slack Connect）**のため、`slack_send_message` での直接投稿はできない（`mcp_externally_shared_channel_restricted` エラーになる）。**必ず `slack_send_message_draft` で下書きを作成**し、ユーザーに手動送信してもらう
- 文面フォーマット（過去の告知メッセージに準拠）:

  ```
  <!here> :tada: 本日 M/D リリースを行いました :tada:

  <今回の目玉機能を1〜2文で要約。あわせて UI 調整や不具合修正にも触れる>
  詳しくはリリースノートをご覧ください。

  https://supergoodmeetings.com/release_notes/YYYYMMDD
  ```

  - 冒頭は `<!here>`（@here の一斉通知）と `:tada:` を添える
  - リリース当日なら「本日 M/D」、前日のリリースなら「昨日 M/D」のように調整する
  - 末尾に本番のリリースノート URL（`https://supergoodmeetings.com/release_notes/YYYYMMDD`）を貼る
- 下書き作成後、下書きを開く URL を報告し、「内容を確認のうえ手動で送信してください（@here の一斉通知になる）」と案内する

## 注意事項

- 既存のリリースノートと文体を揃えること（敬体、完了形）
- ユーザー視点での変化（何ができるようになったか／何が直ったか）を記述する
- 内部実装の話（リファクタリング、ライブラリ更新など）はリリースノートには含めない
- タイトル確定前にファイルを書き込まない
